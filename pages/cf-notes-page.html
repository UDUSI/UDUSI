The CF (Climate and Forecast) Conventions and UDUNITS2

This page provides guidance on using UDUNITS2 with the Climate and Forecast (CF) Conventions, highlighting compatibility aspects and addressing special considerations.

Overview

  • The CF Conventions states that unit strings “…are recognized by UNIDATA’s UDUNITS2 package …”, which in practice means that the unit either has to be present in the XML files, or follow the UDUNITS2 grammar for constructing unit strings from the units present in the XML files.
  • The UDUSI XML files are designed to be drop-in replacements for the original UDUNITS2 XML files.
  • For all unit strings available in the CF Standard Name Table version 89, the results have been compared between using UDUNTS2 with the original XML files and with the UDUSI XML files. The results are identical, with one exception, see next point.

Logarithmic units

  • The CF Standard Name Tables do in fact contain one unit that is not available in the UDUNITS2 standard XML files, namely dB (decibel). However, as SI now includes bel among the “non-SI units accepted for use with SI” it is included in the UDUSI XML files. This means that, with UDUSI XML files, UDUNITS2 is able to correctly parse the unit string “dB” as consisting of the prefix deci and the unit bel, thus resolving an exception that goes back all the way to Table version 2 from 2006.
  • It must be stressed that UDUNITS2 support for handling the two logarithmic units bel and neper is very limited. Any other unit manipulation than what is shown in the previous point (i.e. identifying “dB” as valid unit string) has not been tested and should be expected to fail or give unreliable results until proven otherwise.
  • Attached to the list of non-SI units in Table 8 of the SI Brochure is a footnote stating “In using these units [i.e. “neper” and “bel”] it is important that the nature of the quantity be specified and that any reference value used be specified.“ UDUNITS2, and UDUSI, follows this by specifically defining two targeted units: B_SPL (for “sound pressure level”), where the reference value is 20 micropascal, and BZ (for radar reflectivity of precipitation), where the reference unit is $(10^{-6} m)^3$.

Temperature units

  • An important aspect of UDUNITS2 is that the package does not include functionality to explicitly distinguish between a units referring to a temperature quantity and a unit referring to a temperature difference quantity. This is essential when converting between thermodynamic temperature units (K and °R), and conventional units (°C and °F). This distinction was made explicit in CF-1.11 section 3.1.2.
  • Instead the UDUNITS2 software implements the following heuristic rules:
    • If the unit string represents a simple temperature unit, i.e. only K, °R, °C, °F, or any of their accepted aliases, then any unit conversion will take the scale offset into account. That is, it is assumed that it is a temperature quantity.
    • If the unit string represents a compound unit, e.g. K s-1, °C m-1, then any temperature conversion will not take the scale offset into account. That is, it is assumed that the temperature component of the quantity represents a temperature difference.
  • In general, these rules are applicable and give the desired results. There are however some exceptions detailed in the two points below.
  • In the CF Standard Name Table there a few standard name where UDUNITS2's rules do not apply. These are:
    • square_of_air_temperature and square_of_sea_surface_temperature, which both have canonical unit K2.
    • product_of_air_temperature_and_specific_humidity that has canonical unit K, even though the unit should be considered a compound unit, which according to SI may be shortened as follows: $kg/kg * K = 1 * K = K$.
    • integral_wrt_depth_of_product_of_conservative_temperature_and_sea_water_density and integral_wrt_depth_of_product_of_potential_temperature_and_sea_water_density, which both have canonical unit kg degree_C m-2. This particular case is discusses in Section 3.1.2 of the conventions text.

    In these five cases it is a temperature quantity (and not a temperature difference quantity) that contributes to form the quantity represented by the standard names and its canonical unit.

  • Another important situation where UDUNITS2's rules do not apply is when the temperature quantity has undergone statistical processing that involves forming differences, e.g. standard error or standard deviation.

Time units

In netCDF files and CF Conventions, temporal data variables are classified into two types: time coordinate variables, which define the time axis, and data variables, which store values of a quantity related to time, such as duration, periodicity, or age. For time coordinate variables, the canonical unit must be in the form <unit> since <reference_time>, whereas data variables typically use units in the form <unit> (though there are a few cases where the former form is required).

The <unit>, which is used in both forms, is usually one of second, minute, hour, or day, or their respective symbols or plural forms. Note that UDUNITS2 (and thus UDUSI) also defines the following units:

  • year: Refers is the mean tropical year, defined as the the average period from one vernal equinox to the next, which in UDUNITS2 is defined as $31\ 556\ 925.9747$ s, or $365.242\ 198\ 781\ 25$ days.
  • month: Defined as year/12, i.e., a “tropical_month”.

Neither of these units align with the conventional meaning of “year” and “month” and should therefore be avoided. In terms of Units of Measure, the conventional “year” and “month” refer to the Gregorian calendar and are not useful due to their varying lengths. Cunsequently, it is neither possible nor meaningful to define a unit “gregorian_year” or “gregorian_month”. Additionally, UDUNITS2 defines several other units of time, but these have limited and specialized use.

In the first form of time units, the phrase since identifies the unit as a time coordinate unit. While the UDUNITS2 grammar allows for several alternative phrases to “since”, these alternatives are known to be incompatible with popular software tools. Consequently, it is strongly recommended to use “since” and avoid the alternatives.

The <reference_time> connects the values in the time coordinate variable to an epoch, i.e. a specific point in time on the time axis. Again, the UDUNITS2 grammar allows several alternative ways to write this, but it is advisable to follow the CF Conventions format y-m-d [H:M:S [Z]], where […​] indicates an optional element. Note that both UDUNITS2 and the CF Conventions replace omitted parts with "00". For example if thereference time is give as "2020-01-01" it is understood to mean "2020-01-01 00:00:00 00". Some other standards interpret omitted parts as an idication of limited precision.

The UDUNITS2 software tools allow conversion between between different time units, both with respect to the <unit> part and the <reference_time> part. While the former is often useful, the latter is limited to conversion between date or datetime in the Gregorian calendar. The other calendars defined in the CF Conventions are not supported.
